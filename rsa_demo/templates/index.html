Quick Explanation

  • RSA creates a public key and a private key from two large prime numbers, p and q.
  • Encryption uses the public key, while decryption uses the private key.
  • Digital signatures reverse that direction: the sender signs with the private key, and the receiver verifies with the public key.
  • For large data, RSA does not encrypt the entire file directly. Instead, it protects a session key, while the actual data is encrypted symmetrically.

Technical Details

  • Prime generation: probabilistic Miller-Rabin testing.
  • Encryption padding: OAEP with SHA-256.
  • Signatures: SHA-256 with a simplified PKCS#1 v1.5-style signature block for learning purposes.
  • Secure transfer simulation: RSA wraps the session key, AES-256 CTR encrypts file data, and HMAC-SHA256 checks integrity.

3. Secure File Transfer Simulation

The sender generates a symmetric session key, encrypts the file with AES-256 CTR, and then encrypts the session key with the receiver's RSA public key. The sender's RSA signature is attached to the ciphertext so the receiver can verify it.

Flow Overview

  • 1. Generate RSA key pairs for the sender and the receiver.
  • 2. Generate a 256-bit AES session key.
  • 3. Encrypt the file with AES and protect integrity with HMAC.
  • 4. Encrypt the session key with the receiver's public RSA key.
  • 5. Sign the ciphertext with the sender's private key.
  • 6. Decrypt the session key, verify the signature, and restore the original file.
